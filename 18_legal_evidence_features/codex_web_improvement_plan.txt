MineSafe AI 웹사이트 분석 및 향후 발전 계획
작성일: 2026-07-14

0. 분석 범위와 제한

- 분석 대상: 프로젝트 루트의 app.py
- 분석 방식: Python/AST를 사용하지 않고 rg와 PowerShell로 함수 정의, 화면 문자열, 주요 코드 구간을 텍스트 기반으로 확인했다.
- 이번 작업에서는 app.py, 기존 Python 코드, 평가 파일, Vector DB, chunks를 수정하지 않았다.
- .env와 인증정보 파일은 열지 않았다. app.py에서 민감정보와 관련된 줄은 분석 출력에서 비공개 처리했다.
- app.py는 약 7,666줄, 약 315KB의 단일 Streamlit 파일이다.
- 이 문서는 구현 계획이다. 아래에 제안한 파일은 향후 별도 작업에서 사용자가 whitelist로 허용한 경우에만 생성 또는 수정한다.

1. 현재 기능 구조 요약

1-1. 화면과 운영 모드

- Streamlit 단일 페이지 애플리케이션이다.
- 사용자 모드는 “현장관리자 모드”와 “관리자/개발자 모드”로 나뉜다.
- 현장관리자 메뉴에는 안전 질문 답변, 중대재해처벌법 대응 체크리스트, 위험성평가 초안, 대화 이력, Excel 내보내기/보고서가 있다.
- 관리자 모드에서는 Vector DB 상태, 문서 chunk 수, 모델 및 답변 모드, 질문 시나리오 세트, Gemini 연결 상태를 추가로 확인한다.
- 직접 질문 탭과 질문 시나리오 테스트 탭이 함께 있다.

1-2. RAG 처리 흐름

현재 확인한 기본 흐름은 다음과 같다.

사용자 질문
→ 질문 범위/의도/유형 분류
→ 검색어 확장
→ 다국어 임베딩 모델로 질문 벡터 생성
→ ChromaDB에서 문서, metadata, distance 검색
→ 질문 유형별 선호 문서·키워드 가중치로 재정렬
→ 상위 근거의 문서명, chunk_id, 거리값, 본문으로 context 구성
→ 안정성/Gemini/하이브리드 중 선택한 방식으로 답변 생성
→ 외부 모델 실패 시 안정형 fallback 유지
→ 답변, 근거 표, Evidence Ribbon, 상세 metadata, 추천 증빙자료 표시
→ 필요 시 대화 이력 자동 저장 및 보고서/Excel 내보내기

검색은 단순 벡터 순위만 사용하지 않는다. 발파·불발, 전기, 위험성평가, 분진·보호구, 사고 대응, TBM, 복합위험 등의 질문 유형을 구분하고 선호 문서와 키워드 일치도를 이용해 재정렬한다. 동일 문서의 유사 chunk가 결과를 독점하지 않도록 출처 다양화도 적용한다.

1-3. 답변 생성 구조

- 안정성 모드: 검색 근거를 이용한 체크리스트형 로컬 답변
- 자연어 설명 모드: 검색 근거를 prompt에 넣어 Gemini 설명형 답변 생성
- 하이브리드 모드: 안정형 초안을 먼저 만든 뒤 Gemini로 이유와 현장 적용 설명을 보완
- 범위 밖 질문은 광산 안전관리 범위 안내로 전환
- Gemini 실패 시 검색 근거 기반 안정형 답변으로 되돌아가는 fallback이 있다.

1-4. 근거와 법령 표시

- 결과 화면에 문서명, chunk_id, 거리값, 문서 구분, 근거 상태를 표 형태로 표시한다.
- Evidence Ribbon과 근거 상세 expander가 있다.
- 중대재해처벌법 자료 활용 안내, 법적 단정 금지 문구, 추천 증빙자료, 최신 참고 사례와 뉴스 참고자료를 구분해 보여준다.
- 현재 근거 “표시”는 잘 갖춰져 있으나, 답변의 각 주장과 근거가 실제로 연결되는지를 자동 판정하는 검증 계층은 아직 분명하지 않다.

1-5. 평가와 운영 기능

- 30/65/100/110개 질문 시나리오 세트를 선택할 수 있다.
- 검색 적합성, 근거 기반성, 안전·법령 판단 정확성, 실무성을 각 25점으로 평가한다.
- 기존 평가와 현재 재테스트의 점수 및 검색 근거를 비교한다.
- 자동 평가 요약과 전체 평가 결과, 평가 진행률을 표시한다.
- 법령 체크리스트, 위험성평가 초안, 대화 이력, Excel 내보내기 기능이 있다.

1-6. 현재 구조의 핵심 위험

- UI, CSS, 데이터 읽기/쓰기, 검색, 재정렬, prompt, 외부 모델 호출, 평가, 내보내기가 app.py 하나에 결합되어 있다.
- Streamlit의 상단 실행 방식, st.session_state, st.cache_resource, st.stop 때문에 작은 이동도 실행 순서를 바꿀 수 있다.
- 일부 화면 기능은 JSONL, JSON, XLSX, TSV 파일을 쓸 수 있으므로 테스트에서 실수로 기존 데이터를 바꿀 위험이 있다.
- 외부 모델과 실시간 뉴스는 네트워크 상태에 따라 결과가 달라져 회귀 판단이 어렵다.

2. 현장 조치 실행 모드 추가 가능성

가능성: 가능. 기존의 상황 분류, 즉시 판단, 우선 조치, 체크 항목, 추천 증빙자료, 대화 이력을 재사용할 수 있다.

권장 범위:

- 답변을 읽는 화면에서 끝내지 않고 “조치 시작 → 담당자 지정 → 확인 항목 체크 → 사진/기록 위치 메모 → 재개 승인 확인 → 완료” 상태를 단계별로 관리한다.
- 위험 단계에서는 작업중지, 대피, 출입통제, 보고를 화면 상단에 고정한다.
- 각 조치에 담당자, 시작/완료 시각, 확인자, 미완료 사유를 남긴다.
- AI 답변과 실제 조치 기록을 명확히 구분한다.
- 첫 버전은 파일 첨부보다 텍스트 메모와 상태 전환만 제공하는 것이 안전하다.

예상 수정/생성 파일:

- app_backup_before_action_execution_YYYYMMDD_HHMMSS.py: app.py 수정 직전 로컬 복구용 백업
- app.py: 메뉴 연결과 결과 화면의 “현장 조치로 전환” 진입점
- features/action_execution.py: 단계형 조치 UI와 상태 규칙
- services/action_execution_store.py: 조치 기록 저장/조회
- tests/test_action_execution.py: 상태 전환과 필수 항목 검사
- data/action_execution_history.jsonl: 실행 중 생성되는 운영 데이터 후보이며 Git 대상에서 제외

구현 난이도: 높음
기존 기능 손상 위험: 높음

위험 이유:

- 상태 저장, 새로고침, 중복 저장, 동시 사용자 충돌을 처리해야 한다.
- 잘못된 “완료” 또는 “작업 재개 가능” 표시는 안전사고로 이어질 수 있다.
- 법적 증빙과 AI 보조기록을 혼동하지 않도록 문구와 권한 설계가 필요하다.

3. RAG 근거 검증 패널 추가 가능성

가능성: 매우 높음. 기존 결과 객체에 source, chunk_id, distance, text, rank, intent, rerank 상태가 이미 있어 최소 변경으로 시작할 수 있다.

권장 범위:

- 검색 근거 개수, 고유 문서 수, chunk_id 누락, 빈 본문, 거리값 누락을 자동 점검한다.
- 답변에 언급된 문서명/chunk_id가 실제 검색 결과에 존재하는지 확인한다.
- 법령 조항 번호나 수치 표현이 답변에 있을 때 “근거 본문 재확인 필요” 표시를 낸다.
- 근거 부족, 한 문서 편중, 검색 오류, fallback 사용 여부를 색상과 쉬운 문장으로 설명한다.
- 1차 버전에서는 “사실 검증 완료”라고 단정하지 않고 “형식 및 연결성 점검”이라고 표현한다.

예상 수정/생성 파일:

- app_backup_before_evidence_verification_YYYYMMDD_HHMMSS.py
- app.py: 기존 근거 표 아래 검증 패널 호출부
- services/evidence_validation.py: 부작용 없는 검증 함수
- tests/test_evidence_validation.py: 누락, 편중, 허위 source/chunk 인용, 빈 결과 테스트

구현 난이도: 중간
기존 기능 손상 위험: 낮음~중간

위험 이유:

- 거리값은 모델/컬렉션 설정에 따라 의미와 범위가 달라 단일 임계값으로 “정답/오답”을 판정하면 안 된다.
- 단순 문자열 비교는 표현이 다른 정당한 답변을 오탐할 수 있다.
- 기존 Evidence Ribbon과 중복 UI가 생기지 않도록 한 패널로 통합해야 한다.

4. 관리자 품질 대시보드 추가 가능성

가능성: 높음. 이미 관리자 모드, 평가 진행률, 자동 평가 요약, 재테스트 비교가 있다. 새 대시보드는 기존 데이터를 읽기 전용으로 모아 운영 관점으로 재구성하면 된다.

권장 지표:

- 시나리오 평가 완료율과 평균/최저 점수
- 질문 유형별 검색 적합성 및 근거 기반성
- Vector DB 검색 성공률과 근거 0건 비율
- fallback 비율, Gemini 성공/실패/시간초과 비율
- 동일 문서 편중률, chunk_id 누락률
- 최근 회귀 테스트의 통과/실패와 이전 기준 대비 변화

예상 수정/생성 파일:

- app_backup_before_quality_dashboard_YYYYMMDD_HHMMSS.py
- app.py: 관리자 메뉴와 페이지 라우팅
- features/admin_quality_dashboard.py: 지표와 표 UI
- services/quality_metrics.py: 평가 파일을 읽기 전용으로 집계
- tests/test_quality_metrics.py: 빈 값, 중복 행, 점수 범위, 집계 테스트

구현 난이도: 중간
기존 기능 손상 위험: 중간

위험 이유:

- 기존 평가 파일 스키마가 여러 세트로 나뉘어 있어 열 이름과 결측값 정규화가 필요하다.
- 대시보드가 평가 파일을 쓰지 않도록 읽기 전용 경계를 강제해야 한다.
- LLM 응답 성공률과 답변 품질을 같은 지표로 오해하지 않도록 설명이 필요하다.

5. 긴급 사고 대응 모드 추가 가능성

가능성: 가능하지만 가장 신중해야 한다. 현재 사고 대응 의도 분류와 작업중지·대피·통제·보고 중심의 안정형 답변을 기반으로 전용 화면을 만들 수 있다.

권장 범위:

- 첫 화면에 “인명 보호, 추가 위험 차단, 비상 연락, 현장 보존” 순서를 고정한다.
- 사고 유형별로 최소 질문만 받고, 답변 생성 지연 없이 고정 안전 안내를 먼저 표시한다.
- 외부 모델 호출은 보조 설명에만 사용하고 핵심 긴급 조치는 로컬 규칙으로 표시한다.
- 완료 체크가 아니라 “확인됨/미확인/해당 없음” 상태를 사용한다.
- 정확한 신고 번호, 법정 시간, 책임 판단은 최신 공식 기준 검증 전에는 코드에 하드코딩하지 않는다.
- 네트워크나 Vector DB가 실패해도 핵심 비상 안내가 표시되어야 한다.

예상 수정/생성 파일:

- app_backup_before_emergency_response_YYYYMMDD_HHMMSS.py
- app.py: 긴급 모드 진입 버튼과 라우팅
- features/emergency_response.py: 전용 UI와 단계 흐름
- services/emergency_rules.py: 외부 모델에 의존하지 않는 핵심 안전 규칙
- tests/test_emergency_rules.py: 사고 유형, 빈 입력, DB/API 실패 시 안전 안내 테스트

구현 난이도: 높음
기존 기능 손상 위험: 높음

위험 이유:

- 잘못된 순서, 누락, 지연이 실제 현장 안전에 직접 영향을 줄 수 있다.
- 법령과 연락 체계는 최신성 및 사업장별 차이가 있다.
- 일반 RAG 답변 화면과 다른 실패 안전(fail-safe) 설계가 필요하다.

6. 자동 회귀 테스트 추가 가능성

가능성: 높음. 기존 질문 시나리오와 평가 기준을 변경하지 않고 읽기 전용 입력으로 사용할 수 있다.

권장 3단계:

1) 정적/기본 점검
- app.py 문법 검사
- 금지 경로와 평가 파일의 수정 시각/해시가 테스트 전후 동일한지 확인
- 필수 함수명, 메뉴명, 안전 문구의 존재 확인

2) 결정적 단위 테스트
- 질문 의도/유형 분류
- 검색어 확장
- 재정렬 점수와 출처 다양화
- fallback 답변의 필수 안전 항목
- 근거 검증 규칙

3) 통합 회귀 테스트
- 고정 질문 소수 세트로 안정성 모드 실행
- 검색된 source/chunk_id와 필수 안전 문구 비교
- Gemini 테스트는 결과 문장 일치가 아니라 호출 상태, 근거 연결, fallback 유지 여부만 검사
- 기존 평가 파일에는 쓰지 않고 별도 임시 결과를 사용

예상 생성 파일:

- tests/test_question_classification.py
- tests/test_reranking.py
- tests/test_stable_answer_contract.py
- tests/test_rag_smoke.py
- tests/fixtures/rag_regression_cases.json
- scripts/run_readonly_regression.ps1

app.py 수정 예상: 1단계에서는 없음. 테스트하기 어려운 순수 로직을 모듈로 옮기는 후속 단계에서만 별도 승인 후 수정한다.

구현 난이도: 중간
기존 기능 손상 위험: 낮음

주의:

- 테스트 실행은 기존 TSV/CSV/XLSX를 절대 갱신하지 않아야 한다.
- Vector DB와 chunks는 읽기조차 별도 허용 범위를 확인한다. 초기 회귀 테스트는 코드와 고정 fixture 중심으로 시작한다.
- 표준 라이브러리 unittest로 시작하면 새 패키지가 필요 없다.
- pytest, Playwright 같은 도구가 필요해지면 임의 설치하지 말고 필요성과 버전을 먼저 보고한다.

7. app.py 모듈 분리 필요성과 위험성

필요성: 높음

이유:

- 약 7,666줄 한 파일에 서로 다른 책임이 모여 있어 한 기능의 수정이 다른 화면과 저장 로직에 영향을 주기 쉽다.
- 검색/재정렬처럼 테스트하기 좋은 순수 로직과 Streamlit UI가 분리되면 회귀 테스트가 쉬워진다.
- 법령 체크리스트, 위험성평가, 대화 이력, 내보내기 기능은 독립 모듈 후보가 분명하다.

권장 목표 구조 예시:

- app.py: 설정 조립, 모드 선택, 페이지 라우팅만 유지
- core/question_intent.py: 질문 분류와 검색어 확장
- rag/retrieval.py: 임베딩, Chroma 검색, 재정렬
- rag/prompts.py: context와 prompt 구성
- rag/answering.py: 안정형/Gemini/하이브리드 orchestration
- features/: 체크리스트, 위험성평가, 이력, 내보내기, 신규 화면
- services/: 파일 저장, 품질 집계, 근거 검증
- tests/: 순수 함수와 화면 밖 계약 테스트

위험성: 높음

- Streamlit은 위에서 아래로 다시 실행되므로 import 위치와 초기화 순서가 바뀌면 화면이 달라질 수 있다.
- st.cache_resource에 연결된 모델/DB 객체의 생명주기가 바뀔 수 있다.
- st.session_state 키와 st.stop 위치가 달라지면 페이지 전환이 깨질 수 있다.
- ROOT_DIR 기반 상대 경로가 모듈 파일 위치 기준으로 잘못 계산될 수 있다.
- 함수 이동 과정에서 전역 상수, CSS, 타입, 외부 클라이언트 의존성을 빠뜨릴 수 있다.

안전한 분리 원칙:

- 신규 기능 구현과 모듈 분리를 같은 작업에서 하지 않는다.
- 먼저 회귀 테스트와 현재 화면 스크린샷/동작 체크리스트를 만든다.
- 부작용 없는 순수 함수 한 묶음만 이동하고 테스트한 뒤 멈춘다.
- app.py 수정 직전 규칙에 맞는 백업을 만든다.
- import만 바뀐 단계에서도 현장/관리자 모드, 직접 질문/시나리오 탭, fallback을 모두 확인한다.

10. 가장 먼저 구현할 기능 1개 추천

추천: 자동 회귀 테스트의 1단계인 “비파괴 기준선 테스트”

추천 이유:

- app.py와 현재 웹 기능을 바꾸지 않고 시작할 수 있다.
- 이후 근거 검증 패널, 대시보드, 긴급 모드 개발 때 기존 동작이 깨졌는지 확인할 안전망이 된다.
- 기존 평가 파일에 쓰지 않고 함수명, 메뉴, 안전 문구, 고정 fixture를 확인하도록 만들 수 있다.
- 새 패키지 없이 PowerShell과 표준 라이브러리 unittest 중심으로 설계할 수 있다.

첫 구현 범위는 문법 검사, 필수 구조/문구 검사, 보호 파일 비변경 확인까지만 제한한다. 실제 Vector DB 통합 테스트와 Gemini 테스트는 별도 기능 단계로 분리한다.

11. 단계별 구현 순서

1단계: 비파괴 기준선 회귀 테스트
- app.py 수정 없이 읽기 전용 검사와 fixture 구조를 만든다.
- 보호 파일이 테스트 전후 바뀌지 않았는지 확인한다.

2단계: RAG 근거 검증 패널
- 먼저 순수 검증 함수와 단위 테스트를 만든다.
- app.py에는 기존 근거 표 아래 호출부만 최소 추가한다.

3단계: 관리자 품질 대시보드
- 기존 자동/수동 평가 데이터를 읽기 전용으로 집계한다.
- 저장 기능 없이 조회 화면부터 만든다.

4단계: 순수 로직의 소규모 모듈 분리
- 질문 분류 또는 근거 검증처럼 부작용 없는 한 묶음만 분리한다.
- UI와 저장 로직은 이 단계에서 건드리지 않는다.

5단계: 현장 조치 실행 모드
- 먼저 세션 내 임시 상태로 흐름을 검증한다.
- 영구 저장과 다중 사용자 처리는 별도 후속 단계로 나눈다.

6단계: 긴급 사고 대응 모드
- 공식 안전 담당자 검토를 거친 고정 핵심 안내부터 만든다.
- DB/API 실패 테스트를 통과한 뒤에만 일반 사용자 메뉴에 노출한다.

7단계: 나머지 app.py 모듈 분리
- 한 작업에서 한 모듈만 이동한다.
- 각 이동 후 모든 기준선 테스트와 Streamlit 수동 점검을 반복한다.

12. 테스트 방법

공통 사전 확인:

- 해당 작업의 whitelist를 문서로 확정한다.
- app.py 수정 작업이면 타임스탬프 백업을 먼저 만든다.
- 보호 경로, 기존 평가 파일, 기존 백업의 변경 전 상태를 기록한다.

문법 검사:

- Python 사용이 허용되고 명령이 존재할 때 `python -m py_compile app.py`를 실행한다.
- 프로젝트의 보호 대상 __pycache__에 쓰지 않도록 캐시 경로는 프로젝트 밖 임시 위치로 지정한다.
- 실패하면 추가 수정 전에 오류 줄, 예상 원인, 영향 파일을 보고한다.

Streamlit 수동 점검:

- 현장관리자 모드와 관리자/개발자 모드가 모두 열리는지 확인
- 직접 질문과 시나리오 테스트 탭이 표시되는지 확인
- 안정성 모드에서 고정 질문 3~5개의 답변과 근거가 표시되는지 확인
- 문서명, chunk_id, 거리값, Evidence Ribbon, 상세 보기가 유지되는지 확인
- Vector DB 오류와 외부 모델 오류 시 기존 fallback이 유지되는지 확인
- 체크리스트, 위험성평가, 대화 이력, 내보내기 메뉴가 열리는지 확인하되 기존 평가 저장 버튼은 누르지 않음

기능별 자동 점검:

- 근거 검증: 정상 근거, 빈 근거, source 누락, chunk_id 누락, 한 출처 편중, 허위 인용 fixture
- 품질 대시보드: 빈 파일, 결측 점수, 중복 질문, 점수 범위 밖 값, 세트별 집계
- 현장 조치: 필수 단계 건너뛰기 방지, 중복 저장, 완료 취소, 재개 승인 누락
- 긴급 모드: 빈 입력, DB 없음, API 실패, 시간초과에서도 핵심 안전 안내 유지
- 회귀 테스트: 필수 안전 문구와 구조를 확인하되 LLM 문장 전체 일치는 요구하지 않음

작업 후 확인:

- whitelist 밖 파일의 수정 시각과 내용이 바뀌지 않았는지 확인한다.
- 평가 XLSX/TSV/CSV, Vector DB, chunks, 기존 백업이 변경되지 않았는지 확인한다.
- 생성/수정 파일과 테스트 결과를 한국어로 보고한다.

14. 새 패키지 관련 제안

- 이번 계획 작성에는 새 패키지가 필요하지 않다.
- 1차 회귀 테스트는 PowerShell과 Python 표준 라이브러리 unittest만으로 시작할 수 있다.
- 더 편한 테스트 보고서가 필요하면 pytest, 브라우저 자동화가 필요하면 Playwright를 검토할 수 있으나, 현재 작업에서 설치하지 않는다.
- 패키지 도입은 기능 구현과 분리된 승인 작업으로 진행하고 requirements.txt 변경도 별도 whitelist에 포함해야 한다.

15. 결론

MineSafe AI는 이미 RAG 검색, 질문 유형별 재정렬, 세 가지 답변 모드, fallback, 근거 표시, 법령 대응, 평가와 이력 기능을 갖춘 상태다. 다음 발전의 핵심은 기능 수를 바로 늘리는 것보다 현재 동작을 자동으로 지키는 기준선을 만드는 일이다.

따라서 첫 작업은 app.py를 수정하지 않는 비파괴 회귀 테스트로 한정하고, 그다음 기존 근거 표를 확장하는 RAG 근거 검증 패널을 구현하는 순서를 권장한다. 현장 조치 실행과 긴급 사고 대응은 가치가 크지만 상태 관리와 안전 책임 위험이 높으므로 회귀 테스트, 근거 검증, 관리자 관측 기능이 갖춰진 뒤 단계적으로 진행해야 한다.
