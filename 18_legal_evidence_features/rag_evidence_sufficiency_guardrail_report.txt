7. 기존 RAG 검색 데이터 구조
기존 search_vector_db 함수는 공식 Chroma collection에서 검색한 결과를 dict 목록으로 정리한다. 각 결과에는 문서명 source, chunk_id, 본문 text, 원시 distance, rank, 질문 유형과 의도, 재정렬 진단 정보가 포함된다. 공식 RAG 결과와 사례자료, 네이버 뉴스는 별도 데이터 흐름으로 유지된다.

8. 공식 근거 검색 상태 판정 방식
assess_rag_evidence_sufficiency 함수가 한 질문의 공식 RAG 결과를 한 번만 판정한다. 반환값에는 status, label, reason, 공식 chunk 수, 고유 문서 수, 비어 있지 않은 chunk 수, 중복 결과 수와 관리자용 진단 정보가 포함된다.

세 상태는 다음과 같다.
- 공식 근거 충분: 검색된 공식 문서가 현재 질문의 주요 안전조치를 뒷받침함
- 공식 근거 보완 필요: 관련 공식 문서는 있으나 일부 세부 기준의 추가 확인이 필요함
- 공식 근거 부족: 검색된 공식 문서만으로 구체적인 법령, 수치, 작업 재개 조건을 단정하기 어려움

9. 사용한 기준값
"공식 근거 충분"은 다음 조건을 모두 만족할 때만 부여한다.
- 공식 chunk 3개 이상
- 비어 있지 않은 chunk 3개 이상
- 고유 공식 문서 2개 이상
- 문서명 누락 0개
- chunk_id 누락 0개
- 중복 비율 0 이하
- 단일 문서 편중 비율 0.67 이하

다음 심각한 제한 중 하나라도 있으면 "공식 근거 부족"으로 판정한다.
- 공식 chunk 2개 미만
- 비어 있지 않은 chunk 2개 미만
- 정상 문서명과 chunk_id를 함께 가진 결과 2개 미만
- 고유 공식 문서 1개 미만
- 중복 비율 0.5 초과

충분 조건과 부족 조건 사이의 결과는 "공식 근거 보완 필요"로 분류한다. 질문 의도와 검색 메타데이터의 일치 개수는 진단 정보로만 보존하며 단독 판정 기준으로 사용하지 않는다. 기준값은 상수로 분리해 이후 평가 결과에 따라 조정할 수 있다.

10. ChromaDB score 또는 distance 해석 여부
distance의 방향과 척도는 collection의 실제 metric 설정을 확인하지 않고 단정할 수 없다. 보호 대상인 실제 Vector DB를 열지 않았으므로 distance를 정확도 퍼센트로 바꾸거나 충분성 기준으로 사용하지 않았다. 원시 distance는 기존 검색 결과와 관리자 진단 흐름에만 남고, 이번 판정은 문서와 chunk 구조를 중심으로 수행한다.

11. 정확도 지표가 아닌 이유
chunk 수, 문서 다양성, 식별자 완전성, 중복과 출처 편중은 검색 결과의 구조를 설명할 뿐 답변의 사실 정확성이나 법률상 효력을 증명하지 않는다. 따라서 UI와 코드에서 "공식 근거 검색 상태" 또는 "RAG 근거 충분성 휴리스틱"으로만 표현하며 정확도 점수와 법적 신뢰도 점수로 표시하지 않는다.

12. UI 변경 내용
기존 요약 카드의 단순 결과 개수 기반 "답변 신뢰도"를 "공식 근거 상태"로 교체했다. 기존 answer-runtime-summary CSS를 재사용한 작은 텍스트 배지와 설명을 공식 근거 영역 상단에 표시한다. 색상만으로 구분하지 않고 상태명과 판정 사유를 함께 보여준다. 기존 RAG, KRAS, 증빙자료, 사례와 뉴스 패널의 렌더링 순서와 기능은 유지했다.

13. 현장관리자 모드 표시 내용
현장관리자 기본 화면에는 상태명, "RAG 근거 충분성 휴리스틱" 안내와 짧은 판정 사유만 표시한다. 근거 부족 상태에는 보수적 안전조치와 담당 안전관리자, 관계기관 또는 전문가의 추가 확인 안내를 표시한다. 원시 distance나 내부 진단값은 표시하지 않는다.

14. 관리자·개발자 모드 표시 내용
관리자 모드에서만 닫힌 expander로 다음 내용을 제공한다.
- 공식 chunk 수
- 고유 문서 수
- 비어 있지 않은 chunk 수
- 중복 결과 수
- 문서명과 chunk_id 누락 수
- 판정 사유
- 사용한 기준값
- distance를 충분성 판정에 사용하지 않았다는 해석 방식
인증정보나 환경변수 실제 값은 표시하지 않는다.

15. 근거 부족 시 답변 제한 방식
Gemini와 하이브리드 프롬프트에 공식 RAG 문서명과 chunk_id만 공식 근거로 사용하고, 뉴스와 사례자료는 법령 판단 근거로 사용하지 않으며, RAG에 없는 법령 조항 번호, 수치, 처벌과 작업 재개 조건을 만들지 않도록 지침을 추가했다.

최종 출력 단계에도 안전장치를 적용했다. 공식 근거가 부족하면 생성 답변 대신 작업중지, 접근통제, 대피, 안전관리자 보고, 가스·환기·설비 재점검, 관계기관 또는 전문가 확인, 개선조치 완료와 승인 전 작업 재개 금지 등 보수적 조치만 안내한다. 법 위반이나 처벌을 확정하는 금지 표현은 실제 사용자 답변에 남지 않도록 보수적 문구로 치환한다.

16. 범위 밖 질문 처리

23. 확인하지 못한 사항
보호 규칙에 따라 실제 Vector DB와 chunk 내용을 열지 않았다. 브라우저 상호작용 세션을 만들지 않아 실제 Gemini, 네이버 뉴스, RAG 검색과 다섯 수동 질문의 실데이터 답변은 실행하지 않았다. 네 가지 광산 안전 질문이 기존 분류 키워드와 범위 내 처리 구조로 연결되고, 점심 질문이 evidence보다 앞선 범위 밖 조기 반환으로 분리되는지는 정적 코드 경로로 확인했다.

24. 남아 있는 기술적 한계
- 구조적 휴리스틱은 검색 내용의 법적 정확성을 직접 검증하지 않는다.
- 기준값은 실제 평가 데이터와 전문가 검토 결과에 따라 조정할 필요가 있다.
- 질문 의도와 문서 메타데이터 일치 개수는 현재 진단용이며 충분성 결정에는 직접 사용하지 않는다.
- Streamlit HTTP 확인은 서버 기동 확인이며 실제 외부 API 성공이나 화면별 시각 품질 검증은 아니다.
- 공식 근거가 부족할 때 안전을 우선해 답변을 보수적 공통 조치로 제한하므로 구체성은 낮아질 수 있다.

25. 교수님 설명용 문구
기존 MineSafe AI는 공식 문서명과 chunk_id를 제공했지만, 이번에는 검색된 문서가 현재 질문의 안전조치와 세부 설명을 충분히 뒷받침하는지도 별도로 표시하도록 개선했습니다. 근거가 부족한 경우 법령 조항이나 수치 기준을 임의로 단정하지 않고 보수적인 안전조치와 담당 안전관리자·관계기관의 추가 확인을 안내합니다. 이 상태는 정확도 점수가 아니라 RAG 검색 결과의 구조적 충분성을 나타내는 보조 안전장치입니다.
