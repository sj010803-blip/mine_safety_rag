# MineSafe AI 안전 개발 규칙

## 1. 프로젝트 개요

MineSafe AI는 Streamlit 기반의 광산 안전관리 지원 웹사이트다. 공식 안전 문서와 중대재해처벌법 관련 자료를 ChromaDB에서 검색하고, 검색 근거를 이용해 현장 안전 답변, 체크리스트, 위험성평가 초안, 대화 이력 및 평가 기능을 제공한다.

이 프로젝트는 안전·법령 관련 정보를 다루므로 빠른 변경보다 기존 동작 보존, 근거 확인, 변경 범위 통제가 우선이다. 모든 설명과 작업 보고는 초보자도 이해할 수 있는 한국어로 작성한다.

## 2. 작업 시작 전 허용 범위 확인

1. 파일을 읽거나 수정하기 전에 사용자가 허용한 작업 목표와 경로를 먼저 확인한다.
2. 기존 코드를 수정하기 전 프로젝트 구조, 진입점, 화면 흐름, 데이터 입출력, 외부 서비스 연결, 테스트 위치를 먼저 분석한다.
3. 분석 결과와 예상 변경 파일을 먼저 정리한 뒤 작업한다.
4. 허용 범위가 불명확하면 임의로 넓히지 말고 사용자에게 확인한다.
5. 새 패키지가 필요해 보여도 임의로 설치하지 않는다. 필요한 패키지명, 사용 목적, 영향만 보고한다.

## 3. Whitelist 파일만 수정

1. 매 작업마다 수정 또는 생성이 허용된 파일 목록(whitelist)을 먼저 확정한다.
2. whitelist에 없는 파일은 읽기 분석이 허용된 경우에만 읽고, 수정·이름 변경·이동·삭제하지 않는다.
3. 관련 없는 파일 정리, 자동 포맷팅, import 정렬, 줄바꿈 형식 변경을 하지 않는다.
4. 한 파일을 고치기 위해 다른 파일 변경이 필요해지면 즉시 멈추고 추가 허용을 요청한다.
5. 기존 평가용 `.xlsx`, `.tsv`, `.csv` 파일과 `02_질문시나리오`, `09_answer_tests` 아래의 기존 평가 파일은 별도 명시적 허용 없이는 수정하지 않는다.

## 4. 비밀정보 및 인증정보 접근 금지

1. `.env`를 열거나 내용을 읽거나 출력하지 않는다.
2. API Key, Client ID, Client Secret, 비밀번호, 토큰 또는 기타 인증정보가 들어 있는 파일을 열거나 출력하지 않는다.
3. 비밀값을 검색 결과, 로그, 오류 메시지, 보고서, 테스트 출력에 포함하지 않는다.
4. 인증 설정 확인이 필요하면 값 자체가 아니라 “설정됨/설정되지 않음” 수준으로만 다룬다.

## 5. Vector DB와 chunks 보호

다음 경로는 열어서 내용을 조사하거나 수정·이동·삭제하지 않는다.

- `08_chunks`
- `10_vector_db`
- `10_vector_db_with_major_accident_docs`

또한 `.git`, `__pycache__`, `node_modules`, `.artifact_runtime`을 수정하거나 삭제하지 않는다. Vector DB 상태 확인이 꼭 필요하면 사용자에게 허용 범위를 먼저 확인하고, 기본적으로 애플리케이션 코드에 이미 드러난 연결 구조만 분석한다.

## 6. app.py 수정 전 타임스탬프 백업

1. 향후 작업에서 `app.py` 수정이 허용되더라도 수정 전에 먼저 백업한다.
2. 백업 이름은 다음 형식을 사용한다.

   `app_backup_before_<feature_name>_YYYYMMDD_HHMMSS.py`

3. `<feature_name>`은 영문 소문자와 밑줄을 사용해 변경 목적을 짧고 분명하게 적는다.
4. 백업 생성이 성공했고 원본과 크기가 합리적으로 일치하는지 확인한 뒤에만 `app.py`를 수정한다.
5. 기존 `app_backup_before_*.py`와 그 밖의 기존 백업 파일은 수정하거나 삭제하지 않는다.
6. 백업 파일은 안전 복구용이며 Git 커밋 대상으로 제안하지 않는다.

## 7. 한 번에 하나의 기능만 구현

1. 한 작업에서는 하나의 기능 또는 하나의 버그만 다룬다.
2. 기능 추가와 대규모 리팩터링을 같은 작업에 섞지 않는다.
3. UI 변경, RAG 검색 변경, 프롬프트 변경, 저장 형식 변경을 동시에 진행하지 않는다.
4. 작업 도중 다른 문제를 발견하면 고치지 말고 별도 후속 작업 후보로 보고한다.

## 8. 기존 기능 유지

1. 기존 Streamlit 화면, 현장관리자/관리자 모드, RAG 검색, 안정성/Gemini/하이브리드 답변, fallback, 질문 시나리오 평가, 체크리스트, 위험성평가 초안, 대화 이력 및 내보내기 동작을 임의로 바꾸지 않는다.
2. 기존 함수 이름, 세션 상태 키, 데이터 파일 경로, 저장 형식은 변경 목적에 꼭 필요한 경우에만 다룬다.
3. 안전 관련 문구, 작업중지·대피·출입통제 안내, 근거 문서와 `chunk_id` 표시는 회귀 여부를 반드시 확인한다.
4. 외부 API 실패 시 기존 안정형 답변으로 전환되는 동작을 보존한다.

## 9. 테스트 명령 실행 규칙

1. 테스트는 변경 범위와 직접 관련된 최소 명령부터 실행한다.
2. 명령을 실행하기 전에 평가 파일, Vector DB, chunks 또는 기존 백업을 쓰거나 지우지 않는지 확인한다.
3. `app.py` 문법 확인이 허용되고 Python을 사용할 수 있을 때 기본 명령은 `python -m py_compile app.py`다.
4. 문법 검사 캐시는 보호 대상 프로젝트 경로가 아닌 별도의 임시 캐시 경로를 사용하고 작업 결과에 포함하지 않는다.
5. Python 또는 필요한 도구가 없으면 설치하지 말고 “미실행”과 이유를 보고한다.
6. 문법 검사 통과는 Streamlit 실행, RAG 검색, 화면 동작이 정상이라는 뜻이 아니므로 기능별 최소 확인 항목도 별도로 점검한다.
7. 기존 평가 데이터에 결과를 저장하는 버튼이나 명령은 테스트에 사용하지 않는다.

## 10. 테스트 실패 시 원인부터 보고

1. 테스트가 실패하면 추가 코드를 즉시 수정하지 않는다.
2. 실패 명령, 오류 요약, 영향 범위, 예상 원인, 추가 수정이 필요한 파일을 먼저 보고한다.
3. 원인이 현재 기능 변경인지 환경 문제인지 구분한다.
4. whitelist 밖의 수정이 필요하면 사용자 승인을 받은 뒤 별도 작업으로 진행한다.

## 11. 작업 완료 보고

작업 완료 후 다음 내용을 반드시 보고한다.

1. 생성한 파일 목록
2. 수정한 기존 파일 목록
3. 실행한 테스트 명령과 결과
4. 실행하지 못한 테스트와 이유
5. 금지 파일 및 보호 경로에 접근하거나 변경하지 않았는지 여부
6. 새 패키지를 설치하지 않았는지 여부
7. Git 명령을 실행하지 않았는지 여부

## 12. Git 명령 금지

1. `git commit`, `git push`, `git add`를 실행하지 않는다.
2. 특히 `git add .`와 `git add -A`는 절대 사용하지 않는다.
3. `.git`을 직접 수정하지 않는다.
4. 사용자가 보고서에서 Git whitelist 제안을 요청한 경우에도 커밋 후보 파일명만 제안하고 Git 명령은 실행하지 않는다.
